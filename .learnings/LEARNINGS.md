# Learnings

## 2026-03-14

- Obsidian 设置页在中文研究工作流场景下，不能把长标题、长描述、长路径、长命令和控件同时塞进首页卡片或默认 `Setting` 双栏布局。
- 首页只允许放摘要、状态和入口；所有长路径、长命令、JSON 词表、插件事实明细必须移入右侧详情区或抽屉。
- Companion 插件摘要区不能显示完整配置路径；路径只能在详情区显示，否则会把状态行和按钮排布再次挤坏。
- 桌面宽度“看起来够用”不等于中文 UI 可读。所有仪表盘卡片必须先按中文换行、自适应和行数截断设计，再考虑多列。
- 工作流入口不能继续堆成一整面等高长卡片；必须先按“采集 / 召回 / 整理”分组，再放紧凑动作按钮。
- Hero 区不能把统计信息塞进左列正文里；要拆成“上层摘要 + 下层全宽状态条”，否则右侧控制区会留下大块难看的空白。
- 模块摘要卡禁止使用左右对冲式的 label/value 行；中文场景下必须改为上下堆叠，且桌面端最多两列，避免再次出现挤压和显示不全。
- 如果 Obsidian 设置页里有持续占宽的右侧详情抽屉，而实际窗口宽度又不够，中文内容会在断点之前就先崩坏；这时必须改成顶部标签页的单列结构，而不是继续给双栏补 media query。
- Obsidian 设置页的可用内容宽度往往远小于整个应用窗口宽度，不能再用 viewport `@media` 判断布局切换；必须在设置根容器上启用 `container-type`，并用 container query 控制页签、卡片网格、摘要行和按钮换行。
- CSS Grid 在 Obsidian 设置页里默认会把同一行卡片拉伸到同高，导致短卡出现大片空白、看起来像渲染异常；动作分组、摘要卡和表单卡的网格必须显式 `align-items: start`，必要时让奇数最后一张卡跨满整行。
- 当奇数最后一张工作流分组卡跨满整行时，内部动作列表不能继续保持单列；必须切成两列来吃掉横向空间，否则视觉上仍然像“窗口坏了”。
- 工作流动作按钮不要再挂原生 `title` 提示；在 Obsidian 设置页里会弹出系统级黑色 tooltip，既难看又会让用户误以为还存在渲染错误。
- 工作流页的动作分组卡比普通配置卡更容易在中文场景下挤爆；它的降级断点必须更激进。宁可更早切成单列，也不要为了保留两列而继续压缩按钮文字和说明。
- 不能假设主题的 `button` 基础样式是中性的。工作流动作按钮如果不显式接管 `display / align-items / justify-content / box-sizing / width`，某些主题会把文本视觉居中，造成中文像是“挤出按钮”。按钮文本容器也必须 `width: 100%` 并左对齐。
- 工作流动作按钮不能继续依赖 `button > span` 的轻量结构。真实主题环境里，inline 文本节点很容易被按钮默认样式、行高和对齐规则带偏，出现标题与说明挤在按钮中线附近的错觉；必须改成专用块级文案容器，并显式 reset `appearance / font / text-transform / line-height / overflow`。
- 工作流入口区的降级阈值要独立于普通设置卡。即使其他模块还能维持两列，动作分组也应该更早切成单列，否则中文标题和说明会先崩，再等到通用断点触发就已经太晚。
- Zotero 研究工作流里，`Better BibTeX for Zotero` 不能再被当作隐含背景条件；如果导入链路依赖它，就必须在设置首页、工作流说明和 Zotero 插件事实里明确标成“Zotero 桌面桥接前置条件”。
- 当用户把设置首页当成工作流入口时，首页说明不能只给当前语言的短摘要；必须直接提供中英文双语的详细研究栈、执行顺序和排错说明，避免用户在导入失败时误判成纯布局问题。
